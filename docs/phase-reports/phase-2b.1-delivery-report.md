## 2. Scope

### ما تغيَّر

**Fix 1 — Partial unique indexes on `clients` (the crit bug from the review)**
- `src/db/schema/clients-suppliers.ts`: added two `uniqueIndex` definitions per spec (02_DB_Tree.md + 30_Data_Integrity.md):
  - `clients_name_phone_active_unique` ON (name, phone) WHERE phone != '' AND deleted_at IS NULL
  - `clients_name_email_active_unique` ON (name, email) WHERE email != '' AND deleted_at IS NULL
- `src/db/migrations/0003_clients_dedup_indexes.sql` — generated by drizzle-kit.
- The earlier Phase 2b `createClient` pre-check assumed these existed; they did not. They do now.

**Fix 2 + 3 — Service made duplicate-safe (both create + update)**
- `src/modules/clients/service.ts` rewrote the duplicate-handling layer:
  - `assertNoDuplicate(tx, input, excludeId)` — shared pre-check for BOTH paths. Checks (name, phone) AND (name, email). Excludes `excludeId` so update-on-self doesn't self-collide. Returns 409 `DUPLICATE_CLIENT` with `axis: "phone"|"email"` + `existingId`.
  - `mapUniqueViolation(err, input)` — catches PG 23505 (unique_violation), reads `constraint_name` from the error, maps to the same `ConflictError` code. This is the race-safe last line of defense when two concurrent txs both pass the pre-check.
  - `createClient` now runs `assertNoDuplicate` then wraps INSERT in try/catch → `mapUniqueViolation`.
  - `updateClient` now runs `assertNoDuplicate` WITH `excludeId = id` (the gap the review flagged) then wraps UPDATE in the same 23505 mapping.
- Net: duplicates can no longer sneak through `PUT /api/v1/clients/[id]`, and the "race-only" line from the Phase 2b report is no longer an excuse — the race path also returns 409 with the right shape.

**Fix 4 — Unified validator for user edit (web === API)**
- `src/modules/users/dto.ts`: extracted `UpdateUserPatch` Zod schema (was inlined in the API route). Same schema now exported + imported by both:
  - `src/app/api/v1/users/[id]/route.ts` (API PUT handler).
  - `src/app/(app)/users/[id]/edit/page.tsx` (Server Action).
- The Server Action previously built `patch` manually (`Number()` cast, no bounds check, silent field drops). Now it builds a raw object, feeds it through `UpdateUserPatch.safeParse`, and redirects to `?error=validation&field=...` on failure — matching the new-user form pattern.
- The edit page now accepts `searchParams` + displays the error message, matching the create form UX.

**Fix 5 — Real integration tests for all 4 CRUD paths**
- `tests/integration/users-crud.test.ts` — 7 cases: POST creates + Argon2id hash verified + GET list inclusion; POST duplicate username → 409; POST invalid body → 400; PUT partial patch (name + active); PUT empty patch → 400 (the `refine` rule); PUT missing user → 404; seller POST → 403.
- `tests/integration/clients-crud.test.ts` — 8 cases: POST all fields; POST duplicate (name, phone) → 409 axis=phone; POST duplicate (name, email) → 409 axis=email; POST allows same name when phone + email both empty (partial index doesn't fire); PUT idempotent self-update; **PUT that would collide with ANOTHER client → 409** (the Fix 2 gap); PUT missing → 404; seller POST allowed / seller PUT → 403.
- Both files use `describe.skipIf(!HAS_DB)` — skip cleanly without TEST_DATABASE_URL, run fully when the secret is provisioned.
- `freshRoute()` helpers reset modules + env cache between tests so each case gets a fresh module graph with the mocked session.

### Migration

`0003_clients_dedup_indexes.sql`:
```sql
CREATE UNIQUE INDEX "clients_name_phone_active_unique" ON "clients"
  USING btree ("name","phone")
  WHERE "clients"."phone" <> '' AND "clients"."deleted_at" IS NULL;

CREATE UNIQUE INDEX "clients_name_email_active_unique" ON "clients"
  USING btree ("name","email")
  WHERE "clients"."email" <> '' AND "clients"."deleted_at" IS NULL;
```

**Rollback note**: `DROP INDEX clients_name_phone_active_unique; DROP INDEX clients_name_email_active_unique;` — safe, no data loss.
