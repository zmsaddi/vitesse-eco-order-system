## التفاصيل — كل إجراء وتأثيراته

### 1. إنشاء بيع (POST /api/sales)

| الوحدة المتأثرة | التأثير | النوع | الاتجاه |
|-----------------|---------|-------|---------|
| **العميل** (clients) | إنشاء أو تحديث تلقائي عبر addClient() | تلقائي | أحادي |
| **المنتج** (products) | خصم المخزون بالكمية المطلوبة (حجز فوري) | تلقائي | أحادي |
| **التوصيل** (deliveries) | إنشاء سجل توصيل مرتبط بحالة 'قيد الانتظار' | تلقائي | أحادي |

**المنطق التفصيلي:**
1. قفل صف المنتج (FOR UPDATE) لمنع البيع المتزامن
2. التحقق: الكمية ≤ المخزون الحالي
3. حجز المخزون فورًا (خصم الكمية)
4. إدراج سجل البيع بحالة 'محجوز'
5. إنشاء/تحديث العميل عبر (الاسم + الهاتف) أو (الاسم + الإيميل)
6. إنشاء سجل توصيل مرتبط بـ sale_id

---

### 2. تأكيد التسليم (PUT /api/deliveries → status='تم التوصيل')

| الوحدة المتأثرة | التأثير | النوع | الاتجاه |
|-----------------|---------|-------|---------|
| **البيع** (sales) | تأكيد (status='مؤكد')، تحديث paid_amount/remaining | تلقائي | أحادي |
| **الدفع** (payments) | إدراج سجل تحصيل إذا down_payment_expected > 0 | تلقائي | أحادي |
| **الفاتورة** (invoices) | إنشاء فاتورة برقم تسلسلي (INV-YYYYMM-NNN) | تلقائي | أحادي |
| **العمولة** (bonuses) | حساب وإدراج عمولة البائع + السائق | تلقائي | أحادي |

**المنطق التفصيلي:**
1. تأكيد البيع المرتبط
2. إذا وجدت دفعة مقدمة → إنشاء سجل تحصيل
3. توليد رقم فاتورة ذري (INSERT ON CONFLICT)
4. حساب العمولات حسب user_bonus_rates أو الإعدادات العامة

---

### 3. إنشاء مشترى (POST /api/purchases)

| الوحدة المتأثرة | التأثير | النوع | الاتجاه |
|-----------------|---------|-------|---------|
| **المنتج** (products) | إنشاء إذا جديد؛ تحديث buy_price (متوسط مرجح) + sell_price + stock | تلقائي | أحادي |
| **سجل الأسعار** (price_history) | تسجيل السعر القديم والجديد | تلقائي | أحادي |
| **دفعات الموردين** (supplier_payments) | إنشاء سجل دفعة أولى إذا paidAmount > 0 | تلقائي | أحادي |

**صيغة المتوسط المرجح:**
```
newBuy = (oldStock × oldBuy + qty × price) / (oldStock + qty)
```

---

### 4. إلغاء بيع (POST /api/sales/[id]/cancel)

عملية ذرية من **12 خطوة** تعكس كل التأثيرات:

| الخطوة | الوحدة المتأثرة | التأثير | النوع |
|--------|-----------------|---------|-------|
| 1-4 | **العمولة** | تحقق: هل العمولات مصروفة؟ هل يحتاج قرار؟ | تحقق |
| 5 | **الدفع** | إدراج سجلات استرداد (مبالغ سالبة) لكل تحصيل سابق | تلقائي |
| 6 | **المنتج** | استعادة المخزون (إذا لم يكن ملغيًا مسبقًا) | تلقائي |
| 7 | **العمولة** | حذف أو إبقاء حسب bonusActions | تلقائي |
| 7b | **التسوية** | إنشاء تسويات سالبة لاسترداد العمولات المصروفة | تلقائي |
| 8 | **التوصيل** | تحديث الحالة إلى 'ملغي' | تلقائي |
| 9 | **الفاتورة** | إلغاء ناعم (status='ملغي') أو حذف كامل | تلقائي |
| 10 | **البيع** | تحديث الحالة إلى 'ملغي'، تصفير المدفوع/المتبقي | تلقائي |
| 11 | **الإلغاءات** | تسجيل سجل تدقيق كامل | تلقائي |

**صلاحيات الإلغاء:**
- admin: يلغي أي بيع غير ملغي
- manager: يلغي المحجوز فقط
- seller: يلغي المحجوز الخاص به فقط

---

### 5. تحصيل دفعة (POST /api/sales/[id]/collect أو /api/clients/[id]/collect)

| الوحدة المتأثرة | التأثير | النوع | الاتجاه |
|-----------------|---------|-------|---------|
| **الدفع** (payments) | إدراج سجل تحصيل مع TVA تناسبي | تلقائي | أحادي |
| **البيع** (sales) | تحديث paid_amount, remaining, payment_status | تلقائي | أحادي |

**منطق التحصيل:**
- بيع محدد: POST `/api/sales/[id]/collect`
- FIFO: POST `/api/clients/[id]/collect` → يمشي على المبيعات المفتوحة من الأقدم
- حساب TVA: `tva = amount × vat_rate / (100 + vat_rate)`

---

### 6. إنشاء تسوية (POST /api/settlements)

| الوحدة المتأثرة | التأثير | النوع | الاتجاه |
|-----------------|---------|-------|---------|
| **العمولة** (bonuses) | وسم settled=true + ربط settlement_id (FIFO) | تلقائي | أحادي |

**المنطق:**
- قفل العمولات غير المصروفة (FOR UPDATE) لمنع الدفع المزدوج
- وسم FIFO: العمولات تُصرف بالترتيب (الأقدم أولًا)
- ديون الاسترداد: تُخصم من الرصيد المتاح عند الإلغاء

---

### 7. تغيير سعر المنتج

| المصدر | التأثير | النوع |
|--------|---------|-------|
| مشترى جديد | price_history ← سجل بالسعر القديم والجديد | تلقائي |
| تعديل المنتج (admin) | price_history ← سجل بالسعر القديم والجديد | تلقائي |
| تعديل مشترى | إعادة حساب buy_price من جميع المشتريات | تلقائي |

---

### 8. إدارة المستخدمين

| الإجراء | التأثير | النوع |
|---------|---------|-------|
| إنشاء مستخدم | لا سجل تلقائي في user_bonus_rates (يستخدم الافتراضيات) | — |
| تغيير الدور | يؤثر على حساب العمولة عند التسليم التالي | تفاعلي |
| تعيين تجاوز عمولة | admin يدرج/يحدث في user_bonus_rates | يدوي |
| تعطيل مستخدم | active=false — لا حذف نهائي (BR-37) — السجلات التاريخية تبقى سليمة | يدوي |

## جدول ملخص: جميع الاعتماديات

| الإجراء المصدر | الوحدة المتأثرة | ماذا يتغير | تلقائي؟ | قابل للعكس؟ |
|---------------|-----------------|------------|---------|------------|
| **إنشاء بيع** | العميل | إنشاء/تحديث | نعم | لا يُحذف عند الإلغاء |
| | المنتج | مخزون −كمية | نعم | نعم (عند الإلغاء +كمية) |
| | التوصيل | إنشاء سجل | نعم | نعم (حالة→ملغي) |
| **تأكيد التسليم** | البيع | تأكيد + تحديث الدفع | نعم | نعم (عند الإلغاء) |
| | الدفع | سجل تحصيل | نعم | نعم (سجل استرداد) |
| | الفاتورة | إنشاء + رقم تسلسلي | نعم | نعم (إلغاء ناعم) |
| | العمولة | حساب + إدراج | نعم | نعم (حذف أو إبقاء) |
| **إنشاء مشترى** | المنتج | إنشاء/تحديث buy_price + stock | نعم | نعم (عند الحذف) |
| | سجل الأسعار | سجل تدقيق | نعم | لا (يبقى دائمًا) |
| | دفعات الموردين | سجل دفعة أولى | نعم | نعم (عند حذف المشترى) |
| **إلغاء بيع** | الدفع | سجلات استرداد (−مبلغ) | نعم | — |
| | المنتج | استعادة المخزون | نعم | — |
| | العمولة | حذف أو إبقاء | يدوي | عبر تسوية سالبة |
| | التوصيل | حالة→ملغي | نعم | — |
| | الفاتورة | إلغاء ناعم | نعم | — |
| **تحصيل دفعة** | الدفع | سجل تحصيل | نعم | عند إلغاء البيع |
| | البيع | تحديث paid_amount, remaining | نعم | عند إلغاء البيع |
| **إنشاء تسوية** | العمولة | وسم settled + settlement_id | نعم | عبر تسوية سالبة |
| **تغيير سعر** | سجل الأسعار | سجل تدقيق | نعم | لا (يبقى دائمًا) |

## الأنماط المعمارية الرئيسية

1. **المعاملات الذرية المتسلسلة**: جميع التأثيرات المتعددة تعمل داخل `withTx()` — إما تنجح كلها أو تفشل كلها
2. **الحذف الناعم للتدقيق**: السجلات الملغية تبقى بحالة 'ملغي' ولا تُحذف فعليًا
3. **تسوية FIFO للعمولات**: العمولات تُصرف بالترتيب الأقدم أولًا
4. **ديون الاسترداد**: العمولات المصروفة عند الإلغاء تنشئ تسويات سالبة تُخصم من الرواتب المستقبلية
5. **قفل الصفوف عند التأكيد**: صفوف المنتجات تُقفل (FOR UPDATE) لمنع البيع المتزامن
6. **أهلية العمولة حسب الدور**: فقط seller و driver يكسبون عمولات
7. **المتوسط المرجح**: كل مشترى يعيد حساب buy_price كمتوسط مرجح لجميع المشتريات
