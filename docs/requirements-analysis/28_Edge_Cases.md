# الحالات الاستثنائية — Edge Cases

> **رقم العنصر**: #27 | **المحور**: و | **الحالة**: مكتمل

---

## 1. إلغاء بيع بعد التسليم
- **الحالة**: بيع مؤكد + تم التوصيل + فاتورة + عمولات
- **السلوك**: admin فقط يلغي عبر 12 خطوة ذرية
- **التأثير**: استرداد الدفعات + استعادة المخزون + إلغاء الفاتورة + معالجة العمولات
- **القرار المطلوب**: هل تُحذف العمولات أم تُبقى؟

## 2. إلغاء بيع مدفوع جزئيًا
- **الحالة**: بيع آجل مع تحصيلات سابقة
- **السلوك**: لكل تحصيل سابق → سجل استرداد (مبلغ سالب + TVA سالب)
- **النتيجة**: paid_amount = 0, remaining = 0, payment_status = 'cancelled'

## 3. إلغاء مزدوج (Double Cancel)
- **الحالة**: محاولة إلغاء بيع ملغي مسبقًا
- **السلوك**: خطأ "الطلب مُلغى مسبقاً" — حماية الـ idempotency
- **الوقاية**: فحص الحالة في الخطوة 1 من cancelSale

## 4. العمولات المصروفة عند الإلغاء
- **الحالة**: عمولة settled=true ثم إلغاء البيع
- **السلوك**: خطأ 409 (SETTLED_BONUS_SELLER/DRIVER/BOTH)
- **الحل**: تسوية سالبة (دين استرداد) تُخصم من الرواتب المستقبلية

## 5. حذف مورد له مشتريات
- **الحالة**: مورد مع سجلات مشتريات مرتبطة
- **السلوك المعتمد**: الحذف **ممنوع** — رسالة: "لا يمكن حذف المورد — يوجد مشتريات مرتبطة به"
- **القاعدة**: BR-33 (مطابق لنمط حماية العملاء BR-14)

## 6. حذف عميل له مبيعات
- **الحالة**: عميل مع مبيعات غير ملغية
- **السلوك**: مرفوض — "لا يمكن حذف العميل — يوجد مبيعات مرتبطة به"

## 7. حذف منتج له مخزون أو سجلات
- **الحالة**: منتج stock > 0 أو له مبيعات/مشتريات
- **السلوك**: مرفوض — حماية سجل التدقيق

## 8. حذف مشترى والمخزون أقل من الكمية
- **الحالة**: حذف مشترى بكمية 10 والمخزون الحالي 5
- **السلوك**: مرفوض — "لا يمكن حذف المشترى - المخزون الحالي أقل من كمية المشترى"

## 9. تكرار اسم العميل/المورد
- **الحالة**: إنشاء عميل بنفس الاسم بدون هاتف/إيميل
- **السلوك**: إرجاع `{ ambiguous: true, candidates: [...] }` — النموذج يبقى مفتوحًا لإضافة هاتف/إيميل للتمييز

## 10. الضغط المزدوج (Double-Tap)
- **العمولات**: UNIQUE INDEX على (delivery_id, role) يمنع التكرار
- **الفواتير**: INSERT ON CONFLICT ذري يمنع ترقيم مكرر
- **نموذج VoiceConfirm**: flag `submitted` يمنع الإرسال المزدوج

## 11. تعطيل مستخدم (الحذف النهائي ممنوع)
- **الحالة**: admin يريد إزالة بائع/سائق من النظام
- **السلوك المعتمد**: التعطيل فقط (active=false) — **لا حذف نهائي أبدًا** (BR-37)
- **العمولات**: تبقى وقابلة للتسوية — لا تُحذف ولا تُصفر
- **تسجيل الدخول**: مرفوض (active=false)
- **السبب**: حذف المستخدم يترك عمولات وتسويات يتيمة

## 12. تعديل بيع مؤكد (بعد التسليم)
- **الحالة**: admin يريد تغيير سعر أو كمية بيع تم تسليمه
- **السلوك المعتمد**: **التعديل المباشر ممنوع** — الإجراء الوحيد هو الإلغاء (12 خطوة) ثم إنشاء بيع جديد (BR-32)
- **السبب**: البيع المؤكد مرتبط بفاتورة وعمولات ودفعات — تعديله يخلق تناقضات مالية
- **التأثيرات عند الإلغاء**: عكس عمولة البائع + عمولة السائق + إعادة المخزون (BR-35)

## 13. تعديل بيع قبل التسليم
- **الحالة**: admin/manager/seller يريد تعديل بيع لم يُسلّم بعد
- **السلوك المعتمد**: **مسموح بحرية** — السعر والكمية والمنتج والعميل (BR-31)
- **الملكية**: created_by لا يتغير — البيع يبقى منسوبًا لمنشئه الأصلي
- **العمولات**: لا عمولات قبل التسليم — لذا لا تأثير مالي
- **المخزون**: إذا تغيرت الكمية → يُعاد الحجز القديم ويُحجز الجديد

## 14. بيع بسعر = buy_price بالضبط
- **الحالة**: البيع بسعر التكلفة (ربح = 0)
- **السلوك**: مسموح لـ admin/manager — مرفوض للبائع (لأن sell_price عادةً > buy_price)

## 13. NUMERIC كـ STRING
- **الحالة**: @vercel/postgres يُرجع NUMERIC(19,2) كـ string
- **الخطر**: "5.00" + "3.00" = "5.003.00" بدل 8.00
- **الحل**: parseFloat() قبل أي عملية حسابية
