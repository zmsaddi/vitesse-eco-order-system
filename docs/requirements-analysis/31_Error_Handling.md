# التعامل مع الأخطاء — Error Handling & User Feedback

> **رقم العنصر**: #30 | **المحور**: و | **الحالة**: مكتمل

---

## 1. النمط المركزي (apiError)

```javascript
apiError(err, fallback, status = 400, logPrefix = '')
```

- **أخطاء بالعربي** (الحرف الأول U+0600–U+06FF): تُرسل للمستخدم كما هي
- **أخطاء تقنية**: تُخفى — يُرسل fallback بدلًا منها + تسجيل server-side

---

## 2. رموز HTTP المستخدمة

| الرمز | المعنى | الاستخدام |
|-------|--------|----------|
| 400 | مدخلات خاطئة | فشل التحقق، حقول مفقودة |
| 401 | غير مصادق | لا يوجد token |
| 403 | غير مصرح | الدور لا يملك الصلاحية |
| 404 | غير موجود | سجل غير موجود |
| 409 | تعارض | عمولة مصروفة، بيع ملغي مسبقًا، كيان متكرر |
| 413 | حجم كبير | ملف صوتي > 10MB |
| 428 | شرط مطلوب | BONUS_CHOICE_REQUIRED — يحتاج قرار المدير |
| 429 | تجاوز الحد | voice/process: 10 طلبات/دقيقة |
| 500 | خطأ خادم | خطأ غير متوقع |

## 3. إخفاء البيانات الحساسة

### سعر التكلفة
- **admin/manager**: "سعر البيع (X€) أقل من سعر التكلفة (Y€). لا يمكن البيع بخسارة."
- **seller**: "سعر البيع المُدخَل غير مقبول. يرجى الالتزام بالسعر الموصى أو أعلى."
- **الفرق**: البائع لا يعرف سعر التكلفة الحقيقي أبدًا

### أخطاء قاعدة البيانات
- لا تُكشف أبدًا للمستخدم
- تُسجل في server logs
- يرى المستخدم: "خطأ في العملية" أو ما شابه

## 4. عرض الأخطاء في الواجهة

### Toast Notifications
- **نجاح**: أخضر، 3 ثوانٍ
- **خطأ**: أحمر، 3 ثوانٍ
- **المحتوى**: نص الخطأ من `error.error` أو `error.message`

### أخطاء النماذج (Inline)
- حدود حمراء على الحقول المخالفة
- رسائل تحت الحقل (مثل: "الكمية أكبر من المخزون المتاح")
- تعطيل زر الإرسال عند وجود أخطاء

### نوافذ التأكيد
- عمليات تدميرية تتطلب تأكيد صريح
- الخطأ بعد التأكيد يظهر كـ Toast

## 5. نمط Try/Catch في API Routes

```javascript
try {
  const result = await dbFunction(data);
  return NextResponse.json({ success: true, ...result });
} catch (err) {
  return apiError(err, 'خطأ في العملية', 400, 'PREFIX');
}
```

- جميع الـ routes تتبع هذا النمط
- voice/process يضيف rate-limit check قبل try
